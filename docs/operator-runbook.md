# Operator Runbook

## Goal

Turn Kentik flow evidence into reviewed Forward intent-check candidates, then
create only the missing Kentik-managed checks in Forward.

## Inputs

- Kentik API email and token.
- Kentik Query API JSON from Data Explorer "Show API Call" or a local fixture.
- Forward base URL, user, password/token, and network ID.
- Optional reviewed Kentik-to-Forward location map.

## Standard Workflow

1. Export observed flows and candidate checks:

   ```bash
   npm run kentik:export -- --input fixtures/kentik-topxdata.demo.json
   ```

2. If raw Kentik endpoints do not resolve directly in Forward, generate a map
   template from the target Forward network:

   ```bash
   npm run forward:location-map -- --flows dist/observed-flows.json --out dist/forward-location-map.template.json
   ```

3. Review and edit the location map. The generated map is a starting point, not
   approval to create checks.

4. Rebuild with the reviewed map:

   ```bash
   npm run kentik:export -- --location-map dist/forward-location-map.template.json
   ```

5. Build the dashboard:

   ```bash
   npm run dashboard:build
   npm run dashboard:serve
   ```

6. Dry-run Forward reconciliation:

   ```bash
   npm run forward:import -- \
     --checks dist/forward-intent-checks.json \
     --manifest dist/forward-kentik-manifest.json \
     --report dist/forward-import-dry-run-report.json
   ```

7. Rebuild the dashboard with the dry-run report:

   ```bash
   npm run dashboard:build -- --import-report dist/forward-import-dry-run-report.json
   ```

8. Apply only after review:

   ```bash
   npm run forward:import -- \
     --checks dist/forward-intent-checks.json \
     --manifest dist/forward-kentik-manifest.json \
     --apply \
     --report dist/forward-import-apply-report.json
   ```

9. Re-run dry-run to prove reconciliation:

   ```bash
   npm run forward:import -- \
     --checks dist/forward-intent-checks.json \
     --manifest dist/forward-kentik-manifest.json \
     --report dist/forward-import-post-apply-report.json
   ```

Expected post-apply result: `create: 0`, `unchanged: N`, `changed: 0`,
`stale: 0`.

## Safety Properties

- Default import mode is dry-run.
- `--apply` creates missing checks only.
- Checks are reconciled by exactly one `kentik-key:*` tag.
- Changed and stale checks are report-only.
- Host and device locations are preflighted before apply.
- Data Connector output is optional visibility only; it does not create intent
  checks.

## Common Failures

### Host or device preflight fails

The Kentik evidence does not map to modeled Forward locations. Review the
location map or choose a different target network/snapshot.

### Changed checks appear

The generated check changed since a previous import, or a user edited the check
in Forward. Do not overwrite automatically. Review the `changed` report.

### Stale checks appear

Forward contains Kentik-managed checks that are no longer generated by the
current Kentik evidence. Stale checks are not deleted by this integration.

## Public Data Rule

Do not commit live exports, credentials, or customer-specific maps. Keep those
in ignored local paths such as `dist/`.
